# 智能补偿机制设计

## 背景

### 问题描述

QQ机器人插件请求2条资源时，如果前2条链接中有失效的，会出现：
- 返回1个失效的原始链接 + 1个可用的转存链接
- 用户体验差：点击失效链接无法访问

### 用户期望

始终返回 N 条**可用的**转存链接，如果前 N 条有失效的，自动尝试后面的候选链接。

---

## 解决方案：智能补偿机制

### 核心思路

从"返回前 N 条（含失效）"改为"尽力凑够 N 条可用的"。

### 实现逻辑

```
旧逻辑：
  PanSou 搜索 → 取前 N 条 → 依次转存 → 返回（包含失败的原始链接）

新逻辑：
  PanSou 搜索 → 取全部候选 → 依次转存 → 统计成功数
  ├─ 成功数 >= N → 返回前 N 条成功的
  └─ 成功数 < N → 继续尝试下一条 → 直到凑够 N 条或链接耗尽
```

### 2. 重写 `_deliver_links` 方法

**位置**: `app/services/resource_service.py:156-219`

**新逻辑**：

```python
async def _deliver_links(self, keyword, pan_type, all_links, target_count):
    """智能转存：尽力凑够 target_count 条可用资源"""
    
    results = []
    failed_count = 0
    
    for idx, link in enumerate(all_links, 1):
        result = await self._deliver_single(keyword, pan_type, link, provider)
        
        # 只保留转存成功的（mode="proxy"）
        if result is not None and result.get("mode") == "proxy":
            results.append(result)
            logger.info(f"✅ 转存成功 ({len(results)}/{target_count})")
            
            # 凑够目标数量，提前返回
            if len(results) >= target_count:
                return results
        else:
            failed_count += 1
            logger.warning(f"⏭️ 跳过失效链接 ({failed_count})")
    
    # 链接耗尽，返回所有成功的（可能 < target_count）
    if len(results) > 0:
        return results
    
    # 全部失效，降级返回原始链接（带警告）
    return [
        {
            "title": f"⚠️ {lnk.note or keyword}",
            "url": lnk.url,
            "mode": "direct",
            "warning": "原始链接可能已失效，转存失败"
        }
        for lnk in all_links[:target_count]
    ]
```

**关键改动**：
1. 转存成功才加入结果集
2. 转存失败直接跳过（不返回原始链接）
3. 凑够目标数量提前返回（性能优化）
4. 全部失效时降级返回原始链接（带警告标记）

## 性能影响

### 最好情况
- 前 N 条全部转存成功 → 性能无变化

### 最坏情况
- 前 N 条全部失效，需尝试更多链接 → 响应时间增加

### 优化措施
1. **提前返回**：凑够 N 条立即返回，不继续尝试
2. **日志优化**：记录尝试次数，便于监控
3. **未来优化**：可考虑并发转存（当前串行）

## 未来优化方向

### 1. 并发转存

当前是串行转存，可改为并发（控制并发数）：

```python
# 分批并发转存（每批5个）
batch_size = 5
for i in range(0, len(all_links), batch_size):
    batch = all_links[i:i+batch_size]
    tasks = [self._deliver_single(...) for link in batch]
    batch_results = await asyncio.gather(*tasks)
    
    for result in batch_results:
        if result and result.get("mode") == "proxy":
            results.append(result)
            if len(results) >= target_count:
                return results
```

### 2. 智能排序

根据历史成功率对候选链接排序，优先尝试成功率高的：

```python
# 按历史成功率降序排列
all_links = sorted(all_links, key=lambda x: get_success_rate(x.url), reverse=True)
```

### 3. 缓存失效链接

记录失效链接的 MD5，避免重复尝试：

```python
# Redis 缓存失效链接（TTL 1小时）
if await self._redis.exists(f"prds:invalid:{resource_key}"):
    logger.info("跳过已知失效链接")
    continue
```

## 总结

**核心改进**：从"返回前 N 条（含失效）"改为"尽力凑够 N 条可用的"

**效果**：
- ✅ QQ群用户体验提升：始终返回可用链接
- ✅ 对所有调用方生效（不只是QQ机器人）
- ✅ 符合"交付中台"的产品定位
- ⚠️ 可能增加响应时间（但有提前返回优化）

**适用场景**：
- QQ机器人固定请求2条资源
- Web前端请求少量资源（1-5条）
- 任何需要高质量资源交付的场景
